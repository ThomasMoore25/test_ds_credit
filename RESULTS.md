# RESULTS.md

## extract()
Regex-парсеры: суммы (3 числовых формата + прописью + сокращения тыс./млн./млрд. +
короткие суммы 100 руб. с обязательной валютой), даты (4 формата → ISO, включая
«1 марта 2025» без «г.»), ИНН (10/12 цифр, отсев OCR-мусора), контрагент
(ООО/АО/ПАО/ИП/СПК/КФХ), subject (маркер «Предмет:» / шаблон поставки /
таблица «Наименование» / нумерованный список работ). Числовая сумма требует
контекста («руб/₽/сумма/итого») + защита от реквизитов: ИНН/БИК/р/с/к/с
исключаются, отрицательные суммы → None.
Результат: 6/7 файлов извлекают subject (только scan_ocr = None).

## classify()
Взвешенный keyword scoring + softmax → [0,1], порог top1-top2 = 0.15.
Эксперимент с 8 порогами (0.05–0.50): при <0.15 — OCR-мусор ложно
классифицируется; при >0.15 — реальные документы сваливаются в unknown.
0.15 — единственный порог с 0 ошибок. Результат: 7/7 OK, 14.3% unknown (OCR).

## check_subject()
13 сельхоз-категорий × keyword + rapidfuzz (fuzzy ≥ 80). LLM через LangChain —
опционально, fallback на keyword обязателен. Сильные запрещённые слова
(офисн, клининг, юридическ, маркетинг, ресторан) с приоритетом над разрешёнными.
Белый список сельхоз-контекстов (агроном, ветеринар, КФХ) перебивает запрещённые.
Результат: 8/8 PASS + 6/6 FAIL + 3 EDGE (0% с низкой уверенностью), F1 = 100%.

## Слабые места
1. OCR-мусор: ИНН с буквами отбрасывается, дата восстанавливается не всегда.
2. Сумма прописью: нет дробных конструкций («двадцать пять с половиной тысяч»).
3. Классификация — keyword weights: на новом типе документа все скоры низкие → unknown.
4. FastAPI без аутентификации — для production нужен API key/JWT.
